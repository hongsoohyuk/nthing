# 제품 백로그 (브레인스토밍 대기)

아이디어 단계. 각 항목은 나중에 `superpowers:brainstorming` → spec → 구현 순으로 진행한다.

## 1. 지도 탭 제거 → 등록 플로우의 "만날 위치 선택"으로 전환

**문제의식 (2026-06-05, 사용자):** 하단 가운데 독립 **지도 탭은 실사용 가치가 낮다.** 지도는 오히려 **엔띵(반띵) 등록 시 "만날 위치"를 지도에서 찍을 때** 쓰는 게 자연스럽다.

**현재 상태:**
- `MainLayout` 하단 탭 3개: 홈 / **지도** / 나 (+ FAB → `/splits/new`)
- `Map.tsx`: 현재 위치 중심 + split 핀 + 슬라이드업(BottomSheet). 카카오맵 정상 동작 확인됨(2026-06-05).
- `CreateSplit.tsx`: 위치를 **현재 GPS(`useLocationStore.current ?? DEFAULT_COORDS`)로 자동** 설정 + **"주소" 텍스트 입력**(`address`)만 있음. 지도로 핀 찍는 UI 없음.

**대략의 방향 (미확정, 브레인스토밍 필요):**
- 지도 탭을 없애고 그 자리/플로우를 재구성 (탭 2개 + 중앙 FAB? 다른 탭?)
- 등록 화면에 **지도 미니뷰 + 핀 드래그/탭으로 좌표 선택** 추가 → `latitude/longitude` 결정
- 홈에서 근처 반띵을 지도로 보는 기능은 유지할지/버릴지 결정 (당근처럼 목록 위주면 지도 탭 불필요)
- 역지오코딩으로 주소 자동 채우기 여부

**열린 질문:** 지도 탭 완전 제거 vs 홈 내 "지도 보기" 토글로 흡수? 등록 시 위치 선택 UX(전체화면 지도 vs 인라인 미니맵)?

## 2. 인앱 채팅 (엔띵 성사 후 참여자 간 소통)

**문제의식 (2026-06-05, 사용자):** 엔띵을 만들고 **매칭/참여가 되면, 만나서 교환하기 위한 채팅이 필요**하다. (CLAUDE.md Phase 2의 "인앱 채팅"과 동일 — 사용자가 필요성 재확인)

**현재 상태:**
- 참여(join) + 자동 매칭(WAITING→MATCHED)은 서버에 구현됨. 매칭 후 소통 수단 없음.
- 채팅 채널 단위 미정: split(반띵) 1개당 1채널? 작성자-참여자 1:1? 다자(N분의 1이니 여러 명)?

**열린 질문 (브레인스토밍 필요):**
- 채팅 단위: split당 그룹챗 vs 작성자↔참여자 1:1
- 실시간 방식: WebSocket/STOMP vs 폴링 vs 서드파티(예: Firebase). 서버는 Spring Boot.
- 푸시 연동(이미 FCM 인프라 있음 — 새 메시지 알림)
- 거래 완료/취소 시 채팅 라이프사이클
- MVP 범위: 텍스트만? 사진/위치 공유?

> 참고: Phase 2 항목. 푸시(FCM) 인프라는 이미 코드 완성 상태라 새 메시지 알림에 재사용 가능.
