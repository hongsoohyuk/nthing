# Nthing Phase 2: 푸시 알림 (위치 트리거 + 이벤트)

작성일: 2026-05-29

## 1. Context

Nthing의 핵심 차별점은 **"근처에서 N미터 내 새 반띵 알림"** 이다. Phase 1(클라이언트 마이그레이션 KMP → Vite+React+Capacitor)이 코드상 완성되어, 이제 Phase 2의 첫 기능으로 **푸시 알림**을 구현한다.

현재 상태(2026-05-29 탐색 기준):
- **서버**(`com.onebite.server`): 알림 관련 코드 전무. 도메인은 `auth` / `split` / `upload` / `user`. 신규 `notification` 도메인 필요.
- **DB**: 개발 H2 in-memory, 운영 PostgreSQL+PostGIS(이미 `application-prod.properties` + `V2__add_postgis_geography.sql` 구성됨). 위치 조회는 `SplitLocationQuery` 전략 패턴(H2 Haversine / PostGIS `ST_DWithin`, 프로필 게이팅).
- **모바일**(`mobile/`): Capacitor 8 플러그인 설치됨(app/browser/camera/geolocation/preferences). 푸시 플러그인 미설치.
- **API 명세**(`docs/api-spec.md`): 디바이스 토큰 등록·알림 엔드포인트 없음.
- **`User` 엔티티**: `id/provider/providerId/nickname/profileImageUrl/createdAt` — 위치·디바이스 토큰 필드 없음.

## 2. 결정 사항 (Decisions)

| # | 결정 | 근거 |
|---|------|------|
| D1 | **범위 = 풀 스코프** (공통 인프라 + 이벤트 알림 + 위치 트리거) | Phase 2 푸시의 핵심 가치 전체 구현 |
| D2 | **전송 채널 = FCM 단일** (Firebase가 iOS APNs까지 중계) | iOS도 결국 APNs를 거치므로 FCM 통합이 서버 단순(토큰 1종·전송 1경로). 1인 개발 유지보수 최소. 서버에서 APNs HTTP/2+JWT 직접 구현 회피 |
| D3 | **접근법 A (Lean MVP)** — device 테이블(토큰+위치) + `@Async` 팬아웃 + 딥링크, 인앱 알림함 제외 | 정밀 근접 타겟팅을 최소 코드로. 아웃박스/재시도(B)는 트래픽 생긴 뒤 진화 |
| D4 | **도메인 디커플링 = Spring 애플리케이션 이벤트 + `@TransactionalEventListener(AFTER_COMMIT)` + `@Async`** | split 도메인이 알림을 모르게(느슨한 결합) + 커밋 후에만 발송 + 요청 스레드 비차단 |
| D5 | **dev no-op `LoggingFcmSender`** — Firebase 크레덴셜 없으면 로그만 | 로컬 H2/테스트에서 Firebase 없이 전 기능 빌드·검증 |
| D6 | **위치 트리거 = 서버 사이드 반경 쿼리** (지오해시 토픽 대신) | "~m 근처" 정밀도가 브랜드 핵심. 작성자 제외·거리순 가능. 단 기기 위치 서버 저장 → 프라이버시/법률 항목 갱신 필요(§6) |
| D7 | **푸시 = 네이티브 전용** (iOS/Android), 웹 빌드는 no-op | iOS 웹 푸시 한계 + 네이티브 셸 보유. 웹 FCM은 이번 범위 제외 |

전송 채널 상세(FCM 단일 vs 네이티브 분리): iOS는 두 방식 모두 결국 APNs로 전달된다. 차이는 "누가 APNs에 말을 거느냐"로, FCM 단일은 Firebase가 대신 중계(서버는 FCM 토큰 1종·`send()` 1경로), 네이티브 분리는 서버가 APNs `.p8`로 직접 전송(토큰 2종·전송기 2개). FCM 단일 채택.

## 3. Architecture

### 3.1 데이터 모델

`ddl-auto=validate`이므로 스키마는 Flyway가 관리하고 엔티티는 이와 정확히 일치해야 한다. 마이그레이션 timeline: dev는 `classpath:db/migration`만(→ `V1, V3`), prod는 `classpath:db/migration,classpath:db/migration-prod`(→ `V1, V2, V3, V4`).

#### 공통 `db/migration/V3__push_tables.sql` (H2·PG 공용 plain 컬럼)

```sql
-- devices: 유저-기기-토큰-마지막위치
CREATE TABLE devices (
    id                    BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id               BIGINT       NOT NULL,
    fcm_token             VARCHAR(512) NOT NULL,
    platform              VARCHAR(10)  NOT NULL,        -- IOS | ANDROID
    latitude              DOUBLE PRECISION,             -- nullable (아직 위치 미보고)
    longitude             DOUBLE PRECISION,
    location_updated_at   TIMESTAMP,
    nearby_alerts_enabled BOOLEAN      NOT NULL DEFAULT TRUE,  -- 위치 트리거만 게이팅
    created_at            TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at            TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uk_device_token UNIQUE (fcm_token),       -- 토큰이 자연키 → upsert
    CONSTRAINT fk_device_user  FOREIGN KEY (user_id) REFERENCES users (id)
);
CREATE INDEX idx_device_user ON devices (user_id);

-- notifications: 발송 로그 (디버깅/일일캡/추후 인앱함 토대 — 이번엔 API 미노출)
CREATE TABLE notifications (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id    BIGINT       NOT NULL,                    -- 수신자
    type       VARCHAR(30)  NOT NULL,                    -- NEARBY_NEW_SPLIT|SPLIT_JOINED|SPLIT_MATCHED|SPLIT_CANCELLED
    title      VARCHAR(255) NOT NULL,
    body       VARCHAR(500) NOT NULL,
    split_id   BIGINT,                                   -- 딥링크 타겟
    status     VARCHAR(20)  NOT NULL,                    -- SENT | FAILED
    created_at TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_notif_user FOREIGN KEY (user_id) REFERENCES users (id)
);
CREATE INDEX idx_notif_user_created ON notifications (user_id, created_at);
```

#### 운영 전용 `db/migration-prod/V4__device_postgis.sql` (V2 패턴 복제)

```sql
-- devices에 geography 컬럼 (lat/lng에서 생성). nullable이므로 null 가드.
ALTER TABLE devices ADD COLUMN location geography(POINT, 4326);

UPDATE devices
SET location = ST_SetSRID(ST_MakePoint(longitude, latitude), 4326)::geography
WHERE latitude IS NOT NULL AND longitude IS NOT NULL AND location IS NULL;

CREATE INDEX idx_device_location_geo ON devices USING GIST (location);

CREATE OR REPLACE FUNCTION sync_device_location()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.latitude IS NOT NULL AND NEW.longitude IS NOT NULL THEN
        NEW.location := ST_SetSRID(ST_MakePoint(NEW.longitude, NEW.latitude), 4326)::geography;
    ELSE
        NEW.location := NULL;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_device_location
    BEFORE INSERT OR UPDATE OF latitude, longitude ON devices
    FOR EACH ROW
    EXECUTE FUNCTION sync_device_location();
```

> `Device` 엔티티는 `location` geography 컬럼을 **매핑하지 않는다**(트리거가 관리 — `SplitRequest`가 `location`을 매핑하지 않는 것과 동일). H2에는 `location` 컬럼 자체가 없으므로 엔티티에 두면 validate 실패한다.

### 3.3 서버 알림 파이프라인

#### 새 패키지 `com.onebite.server.notification`

`enum Platform { IOS, ANDROID }`, `enum NotificationType { NEARBY_NEW_SPLIT, SPLIT_JOINED, SPLIT_MATCHED, SPLIT_CANCELLED }`, `enum NotificationStatus { SENT, FAILED }`.

#### 도메인 디커플링 — 이벤트 + after-commit + @Async

`SplitService`는 알림 모듈을 **import하지 않는다.** `ApplicationEventPublisher`로 이벤트만 발행:
- `SplitCreatedEvent(splitId)`
- `SplitJoinedEvent(splitId, joinerUserId)`
- `SplitMatchedEvent(splitId)`
- `SplitCancelledEvent(splitId)`

이벤트 클래스는 `split` 패키지(또는 공용 `event` 패키지)에 둔다(알림이 split에 의존, split은 알림에 비의존 — 의존 방향 일관). `NotificationEventListener`가 `@TransactionalEventListener(phase = AFTER_COMMIT)` + `@Async("notificationExecutor")`로 수신 → split·참여자를 **재조회**(이벤트는 ID만 전달, 핸들러가 자체 읽기 트랜잭션에서 로드) → `NotificationService` 호출.

- **AFTER_COMMIT** → 롤백될 split에 푸시하지 않음.
- **@Async** → 팬아웃이 유저 요청 스레드를 막지 않음.

#### FcmSender 추상화

```kotlin
interface FcmSender {
    fun send(targets: List<PushTarget>): List<SendOutcome>   // PushTarget = (token, PushMessage)
}
// SendOutcome = SUCCESS | INVALID_TOKEN | FAILED (+ token)
```

- `FirebaseFcmSender`: 타겟을 FCM `Message`로 변환 → `FirebaseMessaging.sendEach(messages)`(최대 500/배치, **수신자별 상이 메시지 지원** — NEARBY의 거리 개인화에 필요). 응답에서 `UNREGISTERED`/`INVALID_ARGUMENT` → `INVALID_TOKEN` 매핑. (단일 메시지 다토큰인 이벤트 알림도 동일 경로로 처리.)
- `LoggingFcmSender`: Firebase 크레덴셜 없으면 폴백 — 의도된 발송을 로그만, 전부 SUCCESS 반환. **로컬/테스트에서 Firebase 없이 동작.**
- `FcmConfig`가 `firebase.credentials` 유무로 빈 분기(`@ConditionalOnProperty`/`@ConditionalOnMissingBean`).

#### NotificationService 수신자 로직

두 가지 발송 형태가 있다:
- **NEARBY(수신자별 메시지)** — body의 `{거리}m`가 수신자마다 다르므로 `(device, message)` 쌍을 기기별로 빌드한다.
- **이벤트(공유 메시지)** — N명에게 동일 메시지(거리 없음)를 보낸다.

메서드:
- `notifyNearbyNewSplit(split)` → `DeviceLocationQuery.findActiveNearbyExcludingUser(lat, lng, radiusKm, authorId)` → 일일캡 필터 → **기기별** 거리 계산(Haversine) + 메시지 빌드 → `sendToDevices(쌍 목록)`.
- `notifySplitJoined(split, joinerNickname)` → 작성자(`split.author.id`)의 기기들에 공유 메시지.
- `notifySplitMatched(split)` → 작성자 + 전 참여자.
- `notifySplitCancelled(split)` → 전 참여자.
- 저수준 공통 `sendToDevices(pairs: List<(Device, PushMessage)>)`: notifications 로그행(SENT/FAILED) 저장 → `FcmSender.send`(토큰 묶음) → `INVALID_TOKEN`이면 `deleteByFcmToken`. 이벤트 메서드는 `userIds → 기기 조회 → 동일 메시지로 쌍 구성` 후 이 헬퍼를 호출.

#### DeviceLocationQuery (전략 패턴, `SplitLocationQuery` 복제)

```kotlin
interface DeviceLocationQuery {
    fun findActiveNearbyExcludingUser(lat: Double, lng: Double, radiusKm: Double, excludeUserId: Long): List<Device>
}
```
- `H2DeviceLocationQuery`(`@Profile("!prod")`): 리포지토리 JPQL Haversine. `WHERE nearby_alerts_enabled = true AND latitude IS NOT NULL AND location_updated_at >= :threshold AND user_id <> :excludeUserId` + Haversine ≤ radius.
- `PostgisDeviceLocationQuery`(`@Profile("prod")`): 네이티브 `ST_DWithin(location, ..., :radiusMeters)` + 동일 필터.
- 팬아웃이므로 페이지네이션 없이 `List` 반환.

#### 카피 (payload: `notification{title, body}` + `data{type, splitId}`)

CLAUDE.md 카피톤 표준 반영.

| type | 수신자 | title | body |
|------|--------|-------|------|
| `NEARBY_NEW_SPLIT` | 반경 내 활성 기기 | "근처에 새 반띵이 떴어요" | "{작성자}님이 {거리}m 근처에서 {상품명} {수량}개를 반띵하길 원해요" |
| `SPLIT_JOINED` | 작성자 | "반띵 참여 알림" | "{참여자}님이 회원님의 {상품명} 반띵에 참여했어요" |
| `SPLIT_MATCHED` | 전원 | "반띵 매칭 완료 🎉" | "{상품명} 반띵이 성사됐어요. 만나서 나눠요!" |
| `SPLIT_CANCELLED` | 참여자 | "반띵 취소" | "{상품명} 반띵이 취소됐어요" |

#### 스팸/중복 제어 (NEARBY 한정, 모두 config 노출)

- 반경 `push.nearby.radius-km`(기본 3), **작성자 제외**, `nearbyAlertsEnabled=true` & 위치 not null & `location_updated_at ≥ now - push.nearby.active-window-days`(기본 14).
- 유저당 일일캡 `push.nearby.daily-cap`(기본 10) — notifications 테이블 당일 `NEARBY_NEW_SPLIT` 카운트 초과 수신자 스킵.
- 한 유저의 매칭 기기 여러 대면 전부 발송(폰+태블릿 정상).

#### 에러 처리

- 전송 실패 → `status=FAILED` + 로그. `INVALID_TOKEN`(UNREGISTERED 등) → 기기 행 삭제. **재시도 큐 없음(MVP).**
- async 핸들러 예외는 격리(이미 커밋) — 로그만.
- 권한 거부/토큰 없음 → 해당 유저 미수신, 앱 정상 동작.

### 3.4 모바일 통합 (`mobile/src/features/notifications/`)

**의존성**: `@capacitor-firebase/messaging`(양 플랫폼 FCM 토큰) + 네이티브 설정파일 `GoogleService-Info.plist`(iOS) / `google-services.json`(Android).

**`pushService.ts`**:
- `init()` — `Capacitor.isNativePlatform()` 피처 디텍트, **웹은 no-op.**
- `requestPermissionAndRegister()` — 권한 요청 → 허용 시 FCM 토큰 → `POST /api/devices { fcmToken, platform, lat?, lng? }`.
- 리스너 — `tokenReceived`(갱신) → 재등록 / `notificationActionPerformed`(탭) → `data.splitId` → `navigate('/splits/:id')`.
- `unregister()` — `POST /api/devices/unregister` + 로컬 토큰 삭제.

**통합 지점**:
- **권한 프라이밍** — 기존 `BottomSheet`(Phase 1.5) 재사용해 가치 설명 후 OS 프롬프트. **첫 `/home` 진입 시 1회**(로그인 직후 아님 → 허용률↑). "물어봄" 플래그는 `@capacitor/preferences`.
- **위치 동기화** — `locationStore` 재사용. 위치 갱신(MainLayout 진입 + resume, 이미 배선) 시 좌표로 `POST /api/devices` **디바운스** 재호출(유의미 이동/스테일 시만).
- **탭 라우팅** — 콜드스타트(알림으로 앱 최초 실행) 시 라우터 준비 전이면 네비게이션 **큐잉** 후 준비되면 실행.
- **로그아웃** — `authStore.logout()`에서 `pushService.unregister()`.
- **Profile 토글** — "근처 알림" 스위치 → `POST /api/devices { fcmToken, nearbyAlertsEnabled }`.
- **상태** — 작은 `pushStore`(권한상태/토큰)로 토글·프라이밍 게이트 관리.

## 4. 데이터 흐름 (시나리오)

### S1. 근처 새 반띵 (NEARBY_NEW_SPLIT)
1. 유저 A `POST /api/splits` → `SplitService.create()` 저장 → `publishEvent(SplitCreatedEvent(splitId))`.
2. 트랜잭션 커밋.
3. `NotificationEventListener.onSplitCreated`(AFTER_COMMIT, @Async) 실행 → split 재조회.
4. `notifyNearbyNewSplit(split)` → `DeviceLocationQuery.findActiveNearbyExcludingUser(lat,lng,3km,authorId)`.
5. 일일캡 필터 → 기기별 거리 계산 → notifications(SENT) → `FcmSender.send`.
6. `INVALID_TOKEN` 토큰 → 기기 삭제.
7. 유저 B 기기 푸시 도착 → 탭 → `/splits/:id`.

### S2/S3. 참여 → 매칭 (SPLIT_JOINED / SPLIT_MATCHED)
1. 유저 B `POST /api/splits/{id}/join` → `join()` 저장 → `publishEvent(SplitJoinedEvent(splitId, joinerUserId))`.
2. 정원 충족 시 `status=MATCHED` → 추가로 `publishEvent(SplitMatchedEvent(splitId))`.
3. AFTER_COMMIT → `notifySplitJoined`(작성자) [+ `notifySplitMatched`(전원)].

### S4. 취소 (SPLIT_CANCELLED)
1. 작성자 `PATCH /api/splits/{id}/cancel` → `status=CANCELLED` → `publishEvent(SplitCancelledEvent(splitId))`.
2. AFTER_COMMIT → `notifySplitCancelled`(전 참여자).

### S5. 기기 등록 / 위치 동기화 / 로그아웃
- 로그인 + 첫 홈 진입 → 프라이밍 → 권한 → 토큰 → `POST /api/devices`.
- 위치 갱신(포그라운드) → `POST /api/devices`(좌표).
- 로그아웃 → `POST /api/devices/unregister`.

## 5. 설정 / 운영 (셋업 — 코드 외 문서화)

- **서버**: Gradle `com.google.firebase:firebase-admin`; 서비스계정 JSON을 `firebase.credentials`(env/secret 경로)로 주입(로컬 미설정 → `LoggingFcmSender`); `push.nearby.{radius-km,active-window-days,daily-cap}` 프로퍼티(`application.properties`/`application-prod.properties`).
- **클라**: Firebase 프로젝트 생성 + 설정파일 2종 + **APNs 인증키(.p8) Firebase 콘솔 업로드**(iOS). 환경변수는 `mobile/README`에 기재(.example 미생성 — 사용자 선호).
- **인프라**: Firebase 서비스계정을 시크릿(SSM Parameter/마운트 파일)으로 — `infra/` 후속 작업, 이 플랜에서 빌드하지 않음.

## 6. 프라이버시 / 법률

- **기기 마지막 위치 서버 저장** → 개인정보처리방침에 '기기 위치 수집·저장' 명시 + **위치기반서비스사업자 신고**(CLAUDE.md 비즈니스/법률 검토 항목과 연결). 기존에 split 위치는 이미 저장 중이라 정책 일관이지만, '기기 위치'는 신규 수집 항목이므로 고지 갱신 필요.
- 위치는 **포그라운드에서만** 수집(백그라운드 추적은 Phase 3). 정확도는 앱이 이미 캡처하는 수준 그대로.
- 토큰/위치는 로그아웃 시 삭제(`unregister`), 무효 토큰은 발송 실패 시 자동 정리.

## 7. 테스트 전략

- **서버**:
  - `DeviceService` upsert(신규/갱신)·unregister.
  - `DeviceLocationQuery`(H2 Haversine): 반경 내/밖, 작성자 제외, `nearbyAlertsEnabled=false` 제외, 위치 미보고 제외, active-window 경계.
  - `NotificationService` + fake `FcmSender`: 수신자 정확성(4종), payload, 로그행 생성, `INVALID_TOKEN`→기기 삭제, 일일캡 초과 스킵.
  - 이벤트 배선: split create/join/match/cancel → 올바른 수신자(작성자 제외 포함), `@TransactionalEventListener` AFTER_COMMIT 동작.
  - `DeviceController` 통합 테스트(인증 포함).
- **클라**:
  - `pushService` 모킹 플러그인: 권한 grant→등록 API 호출, deny→미등록, tap→navigate, 웹 no-op.
  - Profile 토글 → API 호출.
- **수동(후속, 코드 아님)**: 실기기 FCM/APNs 실제 도달 — Firebase 프로젝트 + APNs .p8 + 도달 가능 서버 필요(§8).

## 8. 리스크

- **실 전송은 외부 자원 게이팅**: Firebase 프로젝트 + APNs `.p8`(Apple Developer) + 도달 가능한 서버(도메인/HTTPS — Phase 1 미결 항목). 따라서 **end-to-end 실 푸시 검증은 그 운영 항목들 이후**; 이번 이터레이션은 `LoggingFcmSender` + 단위/통합 테스트로 로직을 검증한다.
- **팬아웃 비용**(대규모 동시 활성 유저) → 동기 `@Async` 팬아웃 한계. MVP 규모 OK, 트래픽 시 접근법 B(아웃박스+워커)로 진화.
- **콜드스타트 딥링크**: 알림으로 앱 최초 실행 시 라우터 준비 전 네비게이션 큐잉 필요.
- **멀티기기/토큰 처닝**: upsert-by-token + `INVALID_TOKEN` 정리로 수렴.
- **위치 신선도**: 포그라운드에서만 갱신 → 오래 미접속 유저는 active-window에서 자연 제외(의도된 동작).
